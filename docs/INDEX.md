# ドキュメントインデックス

本ディレクトリ（`docs/`）は、**TransferDesk** プロジェクトの唯一の仕様書群を格納します。

## 読む順番

新規参画者は以下の順序でドキュメントを読み進めることを推奨します。

1. **[00_source_of_truth.md](./00_source_of_truth.md)**
   - プロジェクトの憲法です。目的、スコープ、基本方針が記載されています。
2. **[01_overview.md](./01_overview.md)**
   - アプリケーションの全体像、画面遷移、ユーザーフローを把握します。
3. **[02_tech-stack.md](./02_tech-stack.md)**
   - 採用技術とアーキテクチャの設計意図を理解します。
4. **[03_architecture.md](./03_architecture.md)**
   - システム全体の構成、ディレクトリ構造（FSD）、データフローを理解します。
5. **[04_features.md](./04_features.md)**
   - 各機能の詳細な仕様を確認します。
6. **[05_domain.md](./05_domain.md)**
   - 扱うデータ（エンティティ）とその関係性を理解します。

設計・実装の詳細が必要な場合は、適宜以下を参照してください。

- **[06_ui.md](./06_ui.md)**: UI/UX、コンポーネント設計
- **[07_api.md](./07_api.md)**: API、BFF、通信設計
- **[08_database.md](./08_database.md)**: DB設計、インデックス、マイグレーション

## 優先順位ルール

仕様や実装に矛盾が生じた場合、以下の優先順位で判断します。

1. **`00_source_of_truth.md`**（絶対的な基準）
2. その他の `docs/*.md` ファイル
3. コード内のコメント
4. 実際のコード（実装）

## ドキュメント衝突時の判断基準

- ドキュメントと実装が乖離している場合、**原則としてドキュメントが正**です（実装がバグ、または古い）。
- ただし、実装が意図的な改善でありドキュメントの更新漏れである場合は、チーム合意のもとドキュメントを更新します。

## ドキュメント更新原則

- **Source of Truth が変わった時**：仕様変更時は必ずドキュメントを先に、または同時に更新します。
- **リファクタリング時**：アーキテクチャや設計方針に関わる変更を行った場合は `02_tech-stack.md` `03_architecture.md` 等を更新します。
- **機能追加時**：`04_features.md` および `05_domain.md` に追記します。
